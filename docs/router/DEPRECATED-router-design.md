# Port Router — Design Document

**Version:** 0.6.0-draft  
**Status:** Pre-implementation  
**Scope:** Plug-and-play internal routing system for Go projects following Hexagonal Architecture

---

## Overview

The port router is a dependency broker that lives entirely inside a project's `internal/` folder. It has no entry point of its own — it is imported by the host system and wired at that system's startup. Its job is to give every adapter in a project a single, controlled way to declare what it needs and receive it, without ever importing another adapter directly.

It is designed to be copied into any Go project as-is. Once placed under `internal/router/`, it requires no modification to the router itself. All project-specific configuration happens in two files that live alongside it.

---

## Guiding Principles

**No entry point.** The router does not start itself. It is initialised by whoever owns `main.go` or the equivalent wiring layer in the host system.

**Compiler as gatekeeper.** Port names are typed constants. Raw strings cannot be passed to `RouterRegisterProvider` or `RouterResolveProvider`. Any package outside the module cannot import the router at all because it lives in `internal/`.

**Immutable after startup.** The registry seals itself after wiring is complete. No adapter, plugin, or extension can register a new provider once the system is running.

**Minimal touch surface.** Adding a new port requires one line in the port constants file. Adding a new extension requires one import and one line in the extensions list. The router core never changes.

**Self-contained.** The router has no dependencies outside the Go standard library.

---

## Folder Structure

The router must live in its own subfolder inside whichever `internal/` directory the host project uses. It must not be spread across multiple packages.

Files are divided into two explicit zones. This is not a convention — it is enforced by the policy checker and the integrity system. Misreading a mutable file as frozen or vice versa is a critical mistake.

```
your-project/
└── internal/
    └── router/
        │
        │   ── MUTABLE FILES — safe to edit, add ports and extensions here ──
        │
        ├── doc.go                ← package doc, Package Concerns section
        ├── ports.go              ← PortName constants (the whitelist)
        ├── registry_imports.go   ← imports, var block, routerValidatePortName
        ├── extensions.go         ← the two lists (project-specific wiring)
        │
        │   ── FROZEN FILES — never edited directly, protected by router.lock ──
        │
        ├── registry.go           ← RouterRegisterProvider, RouterResolveProvider, RouterSealRegistry
        ├── extension.go          ← Extension interface, Registry handle, RouterLoadExtensions
        │
        │   ── INTEGRITY ──
        │
        └── router.lock           ← NDJSON, one record per frozen function, committed to source control
```

The frozen files contain functions only — no imports, no var blocks, no type declarations. This is what makes function-level restoration safe and unambiguous. The policy checker enforces this with the `policycheck:frozen-file` annotation and rejects any frozen file that contains non-function declarations.

The mutable files are where all ongoing project work happens. An AI agent operating in normal mode only ever needs to touch these files.

This folder is the only thing that needs to be copied between projects. `extensions.go` is the only file that changes per project — everything else is either stable or regenerated by `--router-lock-update`.

### ports.go

The whitelist. This is the first list. Every port that can ever exist in this project must be declared here as a typed constant. This is the only file where new ports are added.

```go
package router

// PortName is a typed string that prevents raw string values from being
// passed to RouterRegisterProvider or RouterResolveProvider. The compiler enforces this.
type PortName string

// Port constants define every valid port in this project.
// To add a new port: add one line here, then register an implementation
// in extensions.go. No other files need to change.
const (
    PortConfig    PortName = "config"
    PortAuth      PortName = "auth"
    PortDB        PortName = "db"
    PortFormatter PortName = "formatter"
    PortScanner   PortName = "scanner"
)
```

**Rules:**

- One constant per port, no exceptions
- Names are lowercase strings, PascalCase constants
- Removing a constant is a breaking change — deprecate first, remove in a later version
- Constants live here and nowhere else — adapters import this package to reference them

### registry_imports.go — MUTABLE

Owns everything that legitimately changes as the project grows — imports, the var block, and `routerValidatePortName` which gains a new case every time a port is added. This file is never frozen.

```go
package router

import (
    "fmt"
    "sync"
)

// Provider is the type all registered implementations must satisfy.
// Callers type-assert the result of RouterResolveProvider to their expected port interface.
type Provider interface{}

var (
    mu       sync.RWMutex
    registry = map[PortName]Provider{}
    sealed   bool
)

// routerValidatePortName checks the port against the declared constants.
// routerValidatePortName is the runtime half of the compile-time gatekeeper.
// Add a new case here whenever a new PortName constant is added to ports.go.
func routerValidatePortName(port PortName) bool {
    switch port {
    case PortConfig, PortAuth, PortDB, PortFormatter, PortScanner:
        return true
    }
    return false
}
```

**Notes on `routerValidatePortName`:**
This switch must mirror `ports.go` exactly. When a new constant is added to `ports.go`, a corresponding case is added here. This is the one place where a port addition touches two lines in two files — it is intentional and unavoidable without reflection, which would undermine the compile-time guarantees.

### registry.go — FROZEN

Contains frozen functions only. No imports, no var blocks, no type declarations. The `policycheck:frozen-file` annotation instructs the policy checker to reject any non-function declaration found in this file and to verify every function against `router.lock` before any other checks run.

```go
package router

// policycheck:frozen-file
// This file is fully protected by router.lock.
// No imports, no var blocks, no type declarations — functions only.
// Direct modification triggers automatic restoration from router.lock.
// To update legitimately: policycheck --router-lock-update

// RouterRegisterProvider binds a provider implementation to a port name.
// RouterRegisterProvider returns an error if the registry is sealed or the port is already registered.
// RouterRegisterProvider is called exclusively from extension wiring during startup.
func RouterRegisterProvider(port PortName, impl Provider) error {
    mu.Lock()
    defer mu.Unlock()
    if sealed {
        return fmt.Errorf("router: registry is sealed, cannot register %q after startup", port)
    }
    if !routerValidatePortName(port) {
        return fmt.Errorf("router: %q is not a declared port — add it to ports.go first", port)
    }
    if _, exists := registry[port]; exists {
        return fmt.Errorf("router: port %q is already registered", port)
    }
    registry[port] = impl
    return nil
}

// RouterResolveProvider returns the provider registered against a port name.
// RouterResolveProvider requires the caller to type-assert the result to their port interface.
// RouterResolveProvider is safe for concurrent use after RouterSealRegistry has been called.
func RouterResolveProvider(port PortName) (Provider, error) {
    mu.RLock()
    defer mu.RUnlock()
    impl, ok := registry[port]
    if !ok {
        return nil, fmt.Errorf("router: no provider registered for port %q", port)
    }
    return impl, nil
}

// RouterSealRegistry locks the registry against further registration.
// RouterSealRegistry is called once at the end of startup wiring,
// before any goroutines or request handlers are started.
func RouterSealRegistry() {
    mu.Lock()
    defer mu.Unlock()
    sealed = true
}
```

### extension.go — FROZEN

The extension contract. Also a frozen file — the `Extension` interface and `RouterLoadExtensions` must never drift or the entire extension system breaks. Annotated with `policycheck:frozen-file` for the same protection as `registry.go`.

```go
package router

import (
    "context"
    "fmt"
)

// Extension is the interface every plugin must implement.
// It declares what ports it will register and handles its own wiring.
// Extensions are loaded once at startup by RouterLoadExtensions.
type Extension interface {
    // Name returns a human-readable identifier used in error messages and logs.
    Name() string

    // Required declares whether this extension is critical to boot.
    // A required extension failure aborts the entire boot sequence.
    // An optional extension failure emits a RouterError and continues.
    // This also drives test strategy — required extensions use testify/require,
    // optional extensions use testify/assert.
    Required() bool

    // Ports returns the port names this extension intends to register.
    // The router validates each name against the known port list before
    // calling RouterProvideRegistration. An extension declaring a port
    // not in ports.go is rejected at startup.
    Ports() []PortName

    // Consumes optionally declares which ports this extension depends on.
    // When non-nil, the router validates the full dependency graph at startup.
    // Return nil to skip graph validation for this extension.
    Consumes() []PortName

    // RouterProvideRegistration performs the actual provider binding.
    // Implementations call reg.RouterRegisterProvider for each port they declared.
    RouterProvideRegistration(reg *Registry) error
}

// AsyncExtension is an optional extension of Extension for adapters
// that require async initialisation such as database connection pools.
// RouterLoadExtensions detects this interface and calls the async path
// when present, the sync path otherwise.
type AsyncExtension interface {
    Extension
    RouterProvideAsyncRegistration(ctx context.Context, reg *Registry) error
}

// RouterError is the structured error type for all router operations.
// It carries the operation that failed, the port involved, the extension
// name if applicable, and the underlying cause. No router error is ever
// a bare string — callers always have enough context to trace the source.
type RouterError struct {
    Op        string   // operation that failed e.g. "register", "resolve", "boot"
    Port      PortName // port involved, empty if not applicable
    Extension string   // extension name, empty if not applicable
    Err       error    // underlying cause
}

// Error implements the error interface.
// RouterError.Error returns a fully qualified message that identifies
// the operation, port, and extension without requiring the caller to
// unwrap and inspect fields manually.
func (e *RouterError) Error() string {
    if e.Extension != "" && e.Port != "" {
        return fmt.Sprintf("router.%s: extension %q port %q: %v", e.Op, e.Extension, e.Port, e.Err)
    }
    if e.Extension != "" {
        return fmt.Sprintf("router.%s: extension %q: %v", e.Op, e.Extension, e.Err)
    }
    if e.Port != "" {
        return fmt.Sprintf("router.%s: port %q: %v", e.Op, e.Port, e.Err)
    }
    return fmt.Sprintf("router.%s: %v", e.Op, e.Err)
}

// Unwrap allows errors.Is and errors.As to traverse the chain.
func (e *RouterError) Unwrap() error { return e.Err }

// Registry is a thin handle passed to extensions during registration.
// It exposes only RouterRegisterProvider, not RouterResolveProvider or RouterSealRegistry,
// so extensions cannot read or seal the registry during their own wiring.
type Registry struct{}

// RouterRegisterProvider is the Registry method extensions call during their wiring phase.
func (r *Registry) RouterRegisterProvider(port PortName, impl Provider) error {
    return RouterRegisterProvider(port, impl)
}

// RouterLoadExtensions initialises all extensions in order and then seals the registry.
// RouterLoadExtensions is called once from the host system's startup, typically main.go.
// Required extension failures abort boot immediately. Optional extension failures
// are collected as RouterErrors and returned after boot completes — they never
// affect unrelated extensions regardless of boot outcome.
func RouterLoadExtensions(exts []Extension) ([]error, error) {
    reg := &Registry{}
    var warnings []error

    for _, ext := range exts {
        for _, port := range ext.Ports() {
            if !routerValidatePortName(port) {
                err := &RouterError{Op: "boot", Port: port, Extension: ext.Name(),
                    Err: fmt.Errorf("undeclared port — add it to ports.go first")}
                if ext.Required() {
                    return nil, err
                }
                warnings = append(warnings, err)
                continue
            }
        }

        var err error
        if async, ok := ext.(AsyncExtension); ok {
            err = async.RouterProvideAsyncRegistration(context.Background(), reg)
        } else {
            err = ext.RouterProvideRegistration(reg)
        }

        if err != nil {
            re := &RouterError{Op: "boot", Extension: ext.Name(), Err: err}
            if ext.Required() {
                return nil, re
            }
            warnings = append(warnings, re)
        }
    }

    RouterSealRegistry()
    return warnings, nil
}
```

### extensions.go — MUTABLE

**This is the only file that changes per project and the correct place for all new extension and routing work.** It contains the two lists.

**List one** is the port constants — declared in `ports.go`, referenced here implicitly through the extensions.
**List two** is the extension registry — every capability the router loads at startup.

An AI agent adding a new route should only ever touch this file, `ports.go`, and `registry_imports.go`. If an agent proposes changes to any frozen file, that is a policy violation.

```go
package router

import (
    "your-project/internal/adapters/envconfig"
    "your-project/internal/adapters/jwtauth"
    "your-project/internal/adapters/postgres"
    "your-project/internal/adapters/formatter"
    "your-project/internal/adapters/scanner"
)

// extensions is the second list.
// To add a capability: write the extension, add its import above,
// add one line here. Nothing else changes.
var extensions = []Extension{
    envconfig.Extension(),
    jwtauth.Extension(),
    postgres.Extension(),
    formatter.Extension(),
    scanner.Extension(),
}

// RouterBootExtensions wires all extensions and seals the registry.
// RouterBootExtensions is called once from the host system's main or initialisation layer.
// Returns optional warnings from non-required extensions alongside any fatal boot error.
func RouterBootExtensions() ([]error, error) {
    return RouterLoadExtensions(extensions)
}
```

### router.lock — INTEGRITY ARTIFACT

An NDJSON file committed to source control. One record per frozen function. Generated by `policycheck --router-lock-update` and never edited by hand. The policy checker reads this file on every run before any other check executes.

Each line is a self-contained record for one frozen function:

```json
{"file":"internal/router/registry.go","fn":"RouterRegisterProvider","checksum":"e3b0c44298fc1c14...","body":"func RouterRegisterProvider(port PortName, impl Provider) error {\n    mu.Lock()\n    ..."}
{"file":"internal/router/registry.go","fn":"RouterResolveProvider","checksum":"a87ff679a2f3e71d...","body":"func RouterResolveProvider(port PortName) (Provider, error) {\n    mu.RLock()\n    ..."}
{"file":"internal/router/registry.go","fn":"RouterSealRegistry","checksum":"eccbc87e4b5ce2fe...","body":"func RouterSealRegistry() {\n    mu.Lock()\n    defer mu.Unlock()\n    sealed = true\n}"}
```

The checksum is computed from the function body string, not the whole file. This means the file can gain new imports or var declarations in `registry_imports.go` without invalidating frozen function checksums.

On any mismatch the policy checker emits a `[FATAL]`, halts all further checks, performs a surgical restore of only the drifted function body using Go's `go/ast` for position-safe splicing, then reports the result:

```
[FATAL] Router core integrity violation detected

  File:     internal/router/registry.go
  Function: RouterSealRegistry
  Expected: eccbc87e4b5ce2fe...
  Got:      7f8e9d2a1b4c3f6e...

  This file is protected by router.lock. Direct modification by AI agents
  or contributors is not permitted. The function body has been restored
  to its locked state. No other functions were affected.

  If a legitimate change to frozen functions is required:
    1. Discuss and agree on the change explicitly before editing
    2. Make the change manually
    3. Run: policycheck --router-lock-update
    4. Commit router.lock alongside the change with a clear commit message

  Aborting. No further checks will run until integrity is confirmed.
```

The git diff for any `router.lock` change is one line per changed function — making it trivially auditable in code review.

An adapter that needs the config and auth ports declares this at construction time. It resolves through the router, never by importing another adapter directly.

```go
// internal/adapters/scanner/scanner.go

package scanner

import (
    "fmt"
    "your-project/internal/ports"
    "your-project/internal/router"
)

type Adapter struct {
    cfg  ports.ConfigPort
    auth ports.AuthPort
}

// New resolves the scanner's dependencies through the router.
// New fails immediately if either port is missing or the wrong type.
func New() (*Adapter, error) {
    cfgRaw, err := router.RouterResolveProvider(router.PortConfig)
    if err != nil {
        return nil, err
    }
    authRaw, err := router.RouterResolveProvider(router.PortAuth)
    if err != nil {
        return nil, err
    }
    cfg, ok := cfgRaw.(ports.ConfigPort)
    if !ok {
        return nil, fmt.Errorf("scanner: config provider does not implement ConfigPort")
    }
    auth, ok := authRaw.(ports.AuthPort)
    if !ok {
        return nil, fmt.Errorf("scanner: auth provider does not implement AuthPort")
    }
    return &Adapter{cfg: cfg, auth: auth}, nil
}

// Extension returns the router.Extension that registers this adapter.
func Extension() router.Extension {
    return &scannerExtension{}
}

type scannerExtension struct{}

func (e *scannerExtension) Name() string             { return "scanner" }
func (e *scannerExtension) Ports() []router.PortName { return []router.PortName{router.PortScanner} }
func (e *scannerExtension) RouterProvideRegistration(r *router.Registry) error {
    adapter, err := New()
    if err != nil {
        return err
    }
    return r.RouterRegisterProvider(router.PortScanner, adapter)
}
```

## Startup Sequence

```
main.go / wiring layer
    │
    └── warnings, err := router.RouterBootExtensions()
            │
            ├── RouterLoadExtensions(extensions)
            │       │
            │       ├── validate each extension's declared ports against ports.go
            │       ├── for each extension:
            │       │       ├── if Required() true  → failure aborts boot immediately
            │       │       ├── if Required() false → failure collected as warning, continues
            │       │       ├── if AsyncExtension   → RouterProvideAsyncRegistration called
            │       │       └── else                → RouterProvideRegistration called
            │       └── all required extensions passed
            │
            └── RouterSealRegistry()
                    └── registry locked — no further registration possible
                        warnings returned to caller for logging or surfacing
```

## Adding a New Port (Checklist)

When a new capability needs to be routed:

1. Add a `PortName` constant to `ports.go`
2. Add the corresponding `case` to `routerValidatePortName` in `registry_imports.go`
3. Write the port interface in `internal/ports/`
4. Write the adapter that implements it
5. Add `Extension()` to the adapter with `RouterProvideRegistration` and `Required()` implemented
6. Add the import and one line to `extensions.go`

The `Required()` declaration on the extension determines both boot behaviour and which testify package the adapter's tests must use. This is enforced by the policy checker.

Total files touched in the router package: `ports.go`, `registry_imports.go`, `extensions.go`. The frozen files are never touched. The rest of the codebase is unchanged.

---

## Adding a New Extension to an Existing Port

If a port already exists but you want a different implementation (e.g. swap `jwtauth` for `paseto`):

1. Write the new adapter implementing the same port interface
2. Replace the old line in `extensions.go`

One line change. No other files touched.

## Security Properties

| Threat | Mitigation |
|---|---|
| External package registering a provider | `internal/` — compiler rejects the import |
| Raw string passed as port name | Typed `PortName` — compiler rejects it |
| Unknown port name via typed constant | `routerValidatePortName` in `RouterRegisterProvider` — rejected at startup |
| Port shadowing (re-registering a port) | Duplicate check in `RouterRegisterProvider` — returns error |
| Late registration after startup | `RouterSealRegistry` — `RouterRegisterProvider` returns error if called after boot |
| Extension declaring ports it doesn't own | Port declaration validated before `RouterProvideRegistration` is called |
| Partial startup leaving registry in bad state | `RouterLoadExtensions` separates required vs optional — seal only happens after all required extensions pass |
| Optional extension failure taking down unrelated required extensions | Warnings collected independently, required and optional failure paths never interact |
| Silent errors obscuring root cause | `RouterError` struct — every error carries op, port, extension name, and cause. `errors.Is`/`errors.As` traverse the full chain |
| AI agent modifying frozen function bodies | `policycheck:frozen-file` + `router.lock` checksum — `[FATAL]` before any other check |
| Frozen file gaining imports or var blocks | `policycheck:frozen-file` rule rejects non-function declarations — hard error |
| Drift restored incorrectly wiping mutable code | AST-based surgical splice — only the drifted function body is replaced |
| Agent silently regenerating `router.lock` | `--router-lock-update` is an explicit flag unavailable in normal agent check mode |

## What This Is Not

- **Not a dependency injection framework.** There is no reflection, no struct tags, no code generation. Everything is explicit.
- **Not a service locator in the pejorative sense.** Adapters declare their dependencies at construction time, not scattered across their methods.
- **Not a plugin system with dynamic loading.** Extensions are compiled into the binary. The extension list is static code, not a config file read at runtime.
- **Not a replacement for Go interfaces.** The port interfaces themselves live in `internal/ports/` as normal Go interfaces. The router only handles who gets wired to whom.

## CLI Flags

### Dynamic command prefix

Every usage line, error message, and remediation hint in the tool's output uses the name the binary was invoked as — never a hardcoded string. This means the tool adopts whatever command name the host project installs it as in their PATH without any configuration or code change.

```go
import (
    "os"
    "path/filepath"
    "strings"
)

// RouterCommandPrefix returns the name the binary was invoked as.
// RouterCommandPrefix is used throughout all output so usage lines and
// remediation hints always reflect the actual installed command name.
// The .exe suffix is stripped on Windows for clean terminal output.
func RouterCommandPrefix() string {
    base := filepath.Base(os.Args[0])
    return strings.TrimSuffix(base, ".exe")
}
```

Examples of the same tool installed under different names:

```
syntx --graph --from PortAuth
policycheck --router-lock-update
myprojectctl --graph --file internal/adapters/scanner/scanner.go
```

All error messages, fatal outputs, and remediation hints use `RouterCommandPrefix()` at the call site — never a hardcoded `policycheck` anywhere in the codebase. This is enforced as a policy rule: any string literal containing `policycheck` outside of comments and this design document is a violation.

The `--ai` metadata record emits the resolved prefix so an AI agent always knows the correct command to surface to the user:

```json
{"type":"meta","cmd":"syntx","version":"0.5.0","mode":"ai"}
```

### Output mode flags

Two output modes are available. These are mutually exclusive — passing both is an error.

**`--user`**
Human-readable output. Full verbose detail, structured tables via `go-pretty`, colour-coded severity levels, clear run boundaries. Designed for a developer reading output in a terminal. Expands warnings with remediation hints. Graph output renders as an aligned tree. Remediation hints reference the resolved command prefix:

```
[FATAL] Router core integrity violation detected

  File:     internal/router/registry.go
  Function: RouterSealRegistry

  To update legitimately: syntx --router-lock-update
```

**`--ai`**
Concise structured output. NDJSON, one record per violation or graph edge. No colour codes, no padding, no hints. Designed to be consumed as context by an AI agent with minimum token cost. Graph output is filtered NDJSON. Violations include all fields needed for the agent to act but nothing decorative.

```json
{"level":"ERROR","file":"go.mod","rule":"go-version","message":"version must match [1.24.* 1.25.*], got 1.25.4"}
{"level":"WARN","file":"internal/policycheck/core/security.go","rule":"file-size","message":"684 LOC, warn threshold 550"}
{"level":"FATAL","file":"internal/router/registry.go","rule":"integrity","fn":"RouterSealRegistry","message":"checksum mismatch, restored from router.lock"}
```

Both flags also enable debug output — stack traces on panics, extension boot timing, port resolution traces. The difference is formatting only, not content.

## Error Handling

### Router errors are never silent and never bare strings

Every error that originates in the router is a `RouterError` carrying the operation, port, extension name, and underlying cause. `errors.Is` and `errors.As` work through the full chain. Callers always have enough context to trace the source without inspecting raw strings.

### Errors do not cross extension boundaries at runtime

After `RouterSealRegistry` the router is a read-only map. It is not in the call path between extensions at runtime. An error in `PortScanner` has no mechanism to reach `PortAuth` through the router because the router is not involved in that communication. Extensions talk through their port interfaces directly — the router only brokered the initial wiring.

### Boot failure granularity

Extensions declare themselves `Required() bool`. This drives both boot behaviour and test strategy:

| Declaration | Boot failure behaviour | Test package |
|---|---|---|
| `Required() true` | Abort immediately, return fatal `RouterError` | `testify/require` |
| `Required() false` | Collect as warning, continue boot, return after seal | `testify/assert` |

A required extension failure never silently takes down an optional extension — the fatal error is returned immediately and clearly identified by extension name and port. An optional extension failure never takes down a required one — warnings are collected separately and returned alongside a clean boot.

### Advanced tracing is an extension concern

OpenTelemetry spans, correlation IDs, error aggregators, and similar observability tooling are extension concerns. The router provides structured `RouterError` types with enough context to be useful at any level. What a system does with those errors — whether it feeds them to a tracing backend, an error aggregator, or just logs them — is the responsibility of the adapter that catches them. The router does not take a dependency on any observability library.

## Dependency Graph

The router knows the full dependency graph at startup because extensions declare what they provide and optionally what they consume. The graph is surfaced via `--graph` with a `--format` flag and a set of filters designed to keep output minimal — especially important for AI agent sessions where context size is a hard constraint.

### Output format

Default is human-readable text. Pass `--format json` for NDJSON, one record per edge. The NDJSON format matches `router.lock` — same tooling can consume both.

```json
{"from":"PortScanner","to":"PortAuth","via":"internal/adapters/scanner/scanner.go","depth":1}
{"from":"PortScanner","to":"PortConfig","via":"internal/adapters/scanner/scanner.go","depth":1}
{"from":"PortAuth","to":"PortDB","via":"internal/adapters/jwtauth/auth.go","depth":2}
```

### Filters

Without filters the full graph is emitted, which can be enormous in a large system. Filters are the primary interface for AI agent use — an agent should always filter to its immediate neighbourhood.

```
policycheck --graph --from PortAuth
```
Everything that `PortAuth` feeds into. Useful when you have changed an auth implementation and want to know what is affected.

```
policycheck --graph --to PortScanner
```
Everything `PortScanner` depends on. Useful when working on the scanner adapter and needing to know what it is allowed to consume.

```
policycheck --graph --file internal/adapters/scanner/scanner.go
```
Only the edges declared by or pointing to this specific file. The most useful filter for AI agent sessions — the agent gets only what is relevant to the file it is working on, regardless of system size.

```
policycheck --graph --depth 2
```
Limits traversal depth from any starting point. Depth 1 is direct dependencies only. Depth 2 pulls in their dependencies. Most AI sessions never need more than depth 2 to work safely without introducing coupling errors.

Filters compose:

```
policycheck --graph --file internal/adapters/scanner/scanner.go --depth 2 --format json
```

### Design note

The `--file` filter is the safety valve for session context cost. A full graph in a large system could consume an entire context window. A single-file filtered graph at depth 2 is a handful of NDJSON lines. AI agents should default to `--file` and increase scope only if needed.

## Dependencies

### The dependency boundary rule

The router package has zero external dependencies — stdlib only, forever. This is a hard rule, not a guideline. The router is the foundation everything else depends on, so any dependency it takes on becomes a transitive dependency of every project that uses it. That cost is never acceptable.

The rule generalises to the entire system as follows:

- **Router package** — zero external dependencies, stdlib only
- **Each extension** — owns its own dependencies, scoped entirely to that adapter package
- **Host system** — owns only what its own adapters need

This means every library decision is local. If a formatter extension uses `go-pretty` for structured terminal output, that dependency lives in the formatter adapter and leaves with it if the formatter is ever swapped. Nothing bleeds across the boundary. An extension is the correct unit of dependency ownership because it is also the correct unit of capability ownership.

### Current library decisions

**`github.com/jedib0t/go-pretty`** — structured terminal output for the formatter extension. Tables, aligned columns, and clear run boundaries solve the readability problem visible in the current policy checker output where repeated warnings across runs blur together. Scoped to the formatter adapter, not the router or policy engine core.

### Deferred library decisions

**mTLS distribution** — no library decision needed now. When the enterprise system requires it, the extension system handles it natively as two new adapters: a distributor extension that owns cert generation and rotation, and a consumer extension per service that declares a network surface. The router's dependency graph already provides the map of what talks to what, which is exactly what a cert distributor needs. No changes to the router contract are required when that time comes.

All other library decisions are deferred until the audit surfaces actual needs.

All questions from the initial draft have been resolved.

**Panic vs error after seal**
`RouterResolveProvider` panics after `RouterSealRegistry` in both modes — a missing port post-seal is always a programming error. Mode is set at wiring time via `router.SetMode(router.ModeDevelopment)` or `router.SetMode(router.ModeProduction)`. Development mode emits a verbose panic message with remediation hints. Production mode emits a structured log entry before panicking for traceability. Mode is sealed alongside the registry and cannot be changed at runtime. No external config file dependency — the router remains self-contained.

**Consumed port declarations**
Optional. The `Extension` interface has a default no-op implementation for `Consumes()` returning nil, meaning undeclared. When declared, the router validates the full dependency graph at startup. When not declared, that check is silently skipped. No forced ceremony for simple cases.

**Async extension initialisation**
Implemented as an optional second interface rather than changing `RouterBootExtensions` signature. Extensions that need async initialisation implement `AsyncExtension` which embeds `Extension` and adds `RouterProvideAsyncRegistration(ctx context.Context, reg *Registry) error`. `RouterLoadExtensions` detects the interface at runtime and calls the async path if present, the sync path otherwise. Extensions that do not need async are unaffected.

**Dependency graph surfacing**
Structured NDJSON output via `--graph --format json`, with mandatory filter support to keep context cost manageable for AI agent use. Resolved above in full.

**Function-level restore granularity**
Resolved by the frozen/mutable file split. Frozen files contain functions only, making whole-file restoration safe and unambiguous without line number fragility.

**`routerValidatePortName` placement**
Lives in `registry_imports.go` as a mutable function. Never frozen.
